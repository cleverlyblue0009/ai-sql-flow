# Limitations and Threats to Validity
## IEEE Access Submission — AI-SQL-Flow Research

**Date:** 2026-05-27

---

## 1. Construct Validity

### 1.1 Ground-Truth Label Construction

**Limitation:** Ground-truth anomaly labels are generated synthetically using `np.random.default_rng(42)` to place known-count anomaly masks. This means labels are independent of actual data features — a row is "anomalous" by random assignment, not because its features are actually anomalous.

**Impact:** Precision/recall metrics measure how well the system's detections overlap with random labels, not true anomaly detection quality. This inflates recall (the system consistently detects ~93–96% of labeled rows) but understates practical precision (the system flags many rows that happen to not be labeled).

**Mitigation:** This is acknowledged as a research methodology constraint. Future work requires expert-annotated financial datasets. The experimental design is consistent with prior works that similarly use synthetic injection for reproducibility.

### 1.2 Anomaly Rate Discrepancy

**Limitation:** The injected anomaly rates (22–37%) are substantially higher than real-world enterprise anomaly rates (typically 1–5%). This inflates F1 scores for recall-oriented methods and makes precision appear worse than it would be in production.

**Mitigation:** Results are presented alongside FPR and confusion matrices so readers can evaluate system behavior at different operating points.

## 2. Internal Validity

### 2.1 Dataset Generator Design

**Limitation:** Datasets are generated by the same team that designed the detection system. There is a risk of alignment between the data generation patterns and the detection logic (e.g., both using the same random seed, same feature distributions).

**Mitigation:** The ground-truth labels are placed *after* data generation using independent `np.random.default_rng(42)` calls, breaking any direct alignment between feature anomaly patterns and ground-truth labels.

### 2.2 Baseline Selection

**Limitation:** The six baselines (rule_based_only, IsolationForest-only, etc.) are all implemented by the same research team using the same codebase infrastructure. A truly rigorous comparison would use independent third-party baseline implementations.

**Mitigation:** All baseline implementations are published in `run_baseline_comparison.py` for third-party verification.

### 2.3 TF-IDF Threshold Sensitivity

**Limitation:** The cosine similarity threshold (0.85) was set based on the system designers' domain knowledge, not cross-validated against held-out data. Different thresholds yield substantially different precision/recall trade-offs.

**Impact:** The reported FP analysis shows TF-IDF is the dominant FP source. A higher threshold (0.90+) would reduce FP significantly at some cost to recall.

## 3. External Validity

### 3.1 Dataset Domain

**Limitation:** All experiments use synthetic financial datasets (GL accounts, trial balance, journal entries) generated from a single schema template. Real-world financial data includes:
- Regional accounting standard variations (IFRS vs. GAAP)
- ERP-specific field naming (SAP vs. Oracle vs. QuickBooks)
- Multi-currency, multi-entity consolidation structures
- Temporal patterns (fiscal year end anomalies, period-close spikes)

**Implication:** Generalizability to other financial systems or non-financial domains requires additional validation.

### 3.2 Scale Limitations

**Limitation:** Scalability testing reaches 100,000 rows. Enterprise financial databases commonly contain millions of rows (journal entry tables in large corporations may exceed 100M records).

**Mitigation:** The sub-linear scaling exponent (α=0.69) suggests continued favorable scaling. Chunked processing recommendation (§5 of STRESS_TEST_REPORT.md) provides a practical path to million-row scale.

### 3.3 SQL Dialect Coverage

**Limitation:** SQL translation evaluation covers MySQL, PostgreSQL, and SQL Server → Snowflake only. Dialects not covered include Oracle PL/SQL, Teradata SQL, IBM DB2, BigQuery SQL, and Databricks SQL.

**Implication:** Translation confidence of 0.92 applies only to the tested dialects. Oracle-specific constructs (CONNECT BY, ROWNUM, DECODE) require separate validation.

## 4. Statistical Validity

### 4.1 Effect Size Interpretation

**Observation:** Cohen's d values of 9–17 are extraordinarily large by social science standards (d > 0.8 = "large"). This is expected in algorithmic ML comparisons where different methods operate in completely different recall regimes, not in overlapping performance bands.

**Implication:** The large effect sizes should be interpreted as "these are fundamentally different systems" rather than "this system is 9× better."

### 4.2 Single-Platform Testing

**Limitation:** All experiments ran on a single Windows 11 machine (single process, no parallelism). Timing results are not generalizable to multi-core Linux servers or cloud environments.

**Expectation:** Production latencies would be 3–10× lower on optimized server infrastructure with parallelism.

## 6. Future Work Directions

1. **Expert-annotated benchmark dataset** — Collaborate with accounting domain experts to produce labeled financial anomaly datasets (similar to UCI Fraud Detection datasets)
2. **Multi-domain validation** — Test on healthcare billing, logistics, retail transaction data
3. **Adaptive threshold tuning** — Replace fixed TF-IDF threshold with cross-validated domain-specific selection
4. **Oracle/Teradata SQL translation** — Extend rule-based engine with additional dialect patterns
5. **Distributed scalability** — Evaluate chunked+parallel processing on Apache Spark or Dask for million-row datasets
6. **Real-world deployment study** — Deploy in controlled enterprise environment; collect actual anomaly labels from audit outcomes
